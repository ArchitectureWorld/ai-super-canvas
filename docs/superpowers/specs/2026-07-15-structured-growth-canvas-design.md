# 结构化生长画布设计

## 决策

以 TapNow 式节点画布为交互外壳，以植物生长作为对话关系、生命周期和回流的语法。正式界面替换 Gate 0 的固定多面板工作台，但复用其“锚点 → 独立分支 → 结论卡 → 显式回写”的领域闭环。

## 用户可完成的完整流程

```text
播种一个问题
→ 生成或编辑主干活文档
→ 选择主干文字形成芽点
→ 长出独立分支并在底部 Composer 中探索
→ 生成待确认成果卡
→ 预览回写影响并确认
→ 形成新的主干修订
→ 让分支继续生长、休眠或代谢为经验卡
```

页面必须在无账号、无外部 AI、无数据库时仍可直接使用；本阶段通过 `localStorage` 保持状态。浏览器刷新后，主干、分支、成果和视图选择仍可恢复。

## 信息与交互结构

- 顶栏：工作区名称、当前生长视图、本地保存状态和重置示例。
- 无限画布：占据最大空间，默认左至右布局。主干在左中，分支朝右上与右下展开，成果卡位于分支前沿，经验卡位于下方。
- 左侧工具轨道：适配视图、缩放与重置等低频动作。
- 右侧 Inspector：仅在选中对象时出现，展示分支来源、完整消息、成果卡和生命周期动作。
- 底部 Composer：唯一高频输入。上下文芯片必须明确是“主干”“锚点”还是“分支”。

主干正文优先支持文本选择；仅标题栏可拖动节点。选区出现“长出分支”动作，创建后自动聚焦新分支并将 Composer 目标切换至该分支。

## 植物语义与领域映射

| 植物语义 | 领域对象或状态 | 用户可见标签 |
| --- | --- | --- |
| 种子 | 首个 `TrunkRevision` 的起始角色，不新增对象 | 项目起点 |
| 主干 | 可演进的 `TrunkDocument` / 当前 `TrunkRevision` | 主干活文档 |
| 芽点 | `SemanticAnchor` | 锚点 / 生长点 |
| 枝 | `Branch` 及独立 `Message` 上下文 | 分支 |
| 花 | `ConclusionCard.status = ready` | 待确认成果 |
| 果 | `ConclusionCard.status = integrated` | 已回写成果 |
| 休眠 | `Branch.lifecycle = dormant` | 休眠 |
| 腐殖质 | 本阶段的代谢分支摘要投影；持久化代谢记录在后续 Gate 补齐 | 经验沉淀 |

植物语义不使用写实树皮、花瓣、泥土或持续环境动画。所有边必须对应真实来源、派生、回流或代谢关系；所有短动效必须对应真实用户动作。

## 数据和布局边界

- `Conversation/Message` 只追加；`TrunkDocument/Revision` 通过新修订演进。回写绝不改写历史消息。
- `WorkspaceState` 是本阶段领域真源；浏览器画布节点和边只是投影。
- `CanvasLayoutState` 保存选择、视口和用户拖动覆盖，不改变来源关系或主干修订。
- 结构 A 的默认位置由纯函数确定：深度决定 X，同级稳定顺序决定 Y；用户覆盖位置后不因新增节点跳动。
- 成果回写必须是显式操作，并以命令收据避免重复追加。

## 实施范围

本阶段交付：高保真可拖拽/可缩放画布、主干选区锚点、独立分支对话、确定性本地建议、成果卡、Diff 预览、显式回写、休眠/代谢、时间线与本地恢复。

本阶段不交付：图片或文件锚点、真实模型调用、数据库、账号、多用户协作、CRDT、自由手绘、通用白板组件、任意依赖边、完整事件重放、移动端创作和自动代谢。

## 验收

1. 空工作区由中央 Composer 播种；提交后形成主干并将 Composer 停靠底部。
2. 主干文本的非空 Unicode 选区能创建具有精确引文的分支。
3. 分支显示来源、独立消息和明确生命周期；消息不会被回写改变。
4. 成果卡必须先经确认再修改主干；确认后产生新修订，重复点击不重复追加。
5. 休眠降低分支视觉优先级；代谢移出生长主视图并留下经验摘要与可追溯事件。
6. 页面在 1280×800 及以上桌面视口中以画布为主体，刷新后恢复工作区。
